# Integration QA — 통합 정합성 검증

가장 빈번한 런타임 버그는 각 모듈이 개별적으로는 정상이지만 **연결 지점에서 데이터 계약이 어긋나는 것**이다. 정적 코드 리뷰와 `npm run build` 통과는 이를 못 잡는다 — 타입 캐스팅·제네릭이 컴파일러를 통과시키기 때문이다. 따라서 QA의 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"**다.

## 핵심 원칙: 양쪽 동시 읽기
경계면을 검증할 때 한쪽만 읽으면 불일치를 못 본다. 반드시 생산자와 소비자를 **같이 열어** shape을 비교한다.

| 검증 대상 | 왼쪽 (생산자) | 오른쪽 (소비자) |
|----------|-------------|---------------|
| 도메인 함수 반환값 | `getChord()`의 return 타입 | `<ChordDiagram>`의 props 기대 |
| 훅 ↔ 컴포넌트 | `useChords()`의 반환 shape | 컴포넌트의 구조분해 |
| API 응답(있으면) | `fetch`/route 응답 | `fetchJson<T>`의 T |
| 라우팅 | 페이지 파일 경로 | `href`/`navigate` 값 |

## 검증 우선순위
1. **통합 정합성** (최우선) — 경계면 불일치
2. **스펙 준수** — architect 계획의 수용 기준 충족
3. **도메인 정확성** — 기타 코드 데이터의 음악적 유효성 (아래 도메인 체크)
4. **테스트·빌드 통과** — 실제 명령 실행으로 확인

상세 체크리스트가 필요하면 `references/web-coherence-checklist.md`를 읽는다.

## 기타 코드 도메인 정확성 체크
음악 이론상 무효한 데이터는 타입을 통과해도 버그다. 코드 데이터/계산을 검증할 때:
- 보이싱의 현 개수가 악기 현 수(기타=6)와 일치하는가
- 프렛 값이 유효 범위인가 (개방현 0, 뮤트 null, 양의 프렛). 음수 프렛 없음
- 운지 손가락 번호가 1~4 또는 null인가
- 코드 이름(root + 품질)과 실제 구성음이 모순되지 않는가 (예: "Cmaj"인데 E♭ 포함은 의심)
- 다이어그램 현 순서(6번줄↔1번줄)가 일관되게 적용되는가 — 좌우 반전 버그가 흔하다

## 검증 실행 (가정하지 말고 실행)
```bash
npx tsc --noEmit     # 타입 정합성 (캐스팅 우회 의심 시 해당 코드 직접 확인)
npm test             # 단위·컴포넌트 테스트
npm run build        # 프로덕션 빌드
```
실행 출력을 리포트에 인용한다. 환경 문제로 실행 불가하면 "미검증"으로 명시하고 정적 분석으로 대체한다 — 실행 못 한 것을 통과로 보고하지 않는다.

## 점진적 QA
전체 완성 후 1회가 아니라, 각 모듈 완성 직후 즉시 검증한다. 초기 경계면 불일치가 후속 모듈로 전파되어 수정 비용이 커지는 것을 막는다.

## 출력: 검증 리포트
`_workspace/03_qa_report.md`에 작성한다. 통과/실패/미검증을 명확히 구분한다.
```markdown
# QA 리포트 — {대상}
## 통과
- [항목] — 근거(파일:라인 또는 명령 출력)
## 실패
- [항목] — 파일:라인, 문제, 수정 방법
## 미검증
- [항목] — 이유
```
실패 항목은 **즉시 해당 에이전트에게** SendMessage로 구체적 수정 요청(파일:라인 + 방법)을 보낸다. 경계면 이슈는 생산자·소비자 양쪽 모두에게 알린다.

## 원칙
- 빌드 통과를 정상 동작으로 착각하지 않는다.
- 약한 체크("타입이 정의됐는가")가 아니라 강한 체크("생산자 shape과 소비자 타입이 일치하는가")를 한다.
- 상충하는 결과는 삭제하지 않고 출처와 함께 병기한다.
