# 웹앱 통합 정합성 체크리스트

`integration-qa` 스킬의 상세 레퍼런스. 경계면을 교차 검증할 때 항목별로 양쪽 코드를 동시에 열어 확인한다.

## 데이터 계약 (컴포넌트/훅/도메인 경계)
- [ ] 도메인 함수의 반환 타입과 소비 컴포넌트의 props 기대가 일치
- [ ] 커스텀 훅의 반환 shape과 컴포넌트의 구조분해가 일치 (필드명·옵셔널 여부)
- [ ] `null`/`undefined` 반환 케이스를 소비자가 처리 (예: 미존재 코드 조회)
- [ ] 배열을 기대하는 곳에 래핑 객체(`{ items: [] }`)가 전달되지 않는지
- [ ] 옵셔널 필드의 null/undefined 처리가 생산자·소비자 양쪽에서 일관

## 타입 안전성 우회 탐지
- [ ] `any` 사용처가 경계면을 가리고 있지 않은지
- [ ] `as` 단언이 실제 런타임 shape과 어긋나지 않는지
- [ ] 제네릭(`fetchJson<T>`)의 T가 실제 응답과 일치하는지 (컴파일러는 못 잡음)
- [ ] camelCase ↔ snake_case 변환이 경계마다 일관 적용

## API 연결 (백엔드/외부 데이터가 있을 때)
- [ ] 모든 API 엔드포인트에 대응하는 호출부(훅/fetch)가 존재하고 실제 호출됨
- [ ] 응답 래핑(`{ data: [...] }`)을 소비자가 unwrap하는지
- [ ] 즉시 응답과 비동기 최종 결과의 shape 차이를 프론트가 구분하는지

## 라우팅 정합성 (라우터 사용 시)
- [ ] 코드 내 모든 링크/네비게이션 값이 실제 존재하는 페이지 경로와 매칭
- [ ] 동적 세그먼트가 올바른 파라미터로 채워지는지
- [ ] 중첩 경로 접두사 누락이 없는지

## 상태/데이터 흐름
- [ ] 상태 기반 분기(`if status === 'X'`)의 X가 실제 도달 가능한 값인지
- [ ] 정의된 상태 전이가 코드에서 실제 실행되는지 (죽은 전이 없음)
- [ ] 도메인 모델 → UI 표현으로의 매핑이 정보 손실/왜곡 없이 이뤄지는지

## 기타 코드 도메인 (자세히)
- [ ] 보이싱 현 개수 = 6 (기타)
- [ ] 프렛 값: 개방현 0, 뮤트 null, 양의 정수만. 음수·비정수 없음
- [ ] 운지 손가락: 1~4 또는 null
- [ ] 다이어그램 현 순서(6→1번줄)가 데이터·렌더 양쪽에서 일관 (좌우 반전 주의)
- [ ] baseFret과 표시 프렛 범위가 보이싱의 실제 프렛을 포함하는지
- [ ] 코드 이름의 root/품질과 구성음이 모순되지 않는지
