# Feature Architecture — 기타 코드 웹앱 기능 설계

기능 요구를 구현자가 모호함 없이 따라갈 수 있는 설계 계획으로 변환한다. 좋은 설계의 핵심은 **데이터 계약을 먼저 확정**하는 것이다 — 컴포넌트 간 주고받는 데이터의 타입이 모호하면 구현 후 경계면 버그로 이어진다.

## 절차

### 1. 컨텍스트 파악
- `package.json` 존재 여부 확인. 없으면 **그린필드** — 스택과 디렉토리 구조를 먼저 제안한다(아래 "그린필드 셋업").
- 있으면 기존 디렉토리 구조, 타입 정의(`src/types`, 도메인 모델), 컨벤션을 읽고 그것을 따른다.

### 2. 요구를 수용 기준으로 분해
기능 요구를 검증 가능한 수용 기준 목록으로 쪼갠다. 각 기준은 "무엇이 참이어야 하는가"를 객관적으로 진술한다.

예: "코드 다이어그램 표시" →
- 코드 이름을 입력하면 해당 코드의 프렛/핑거 포지션이 SVG로 렌더된다
- 존재하지 않는 코드를 입력하면 에러 메시지가 표시된다
- 6번줄(저음)이 다이어그램 좌측, 1번줄(고음)이 우측에 온다

### 3. 데이터 모델 설계 (도메인 타입 먼저)
기타 코드 도메인의 핵심 엔티티를 TypeScript 타입으로 확정한다. **음악 이론상 불가능한 상태가 타입으로 표현되지 않도록** 한다.

핵심 엔티티(프로젝트에 맞게 조정):
```typescript
// 음이름 — 가능한 값을 유니온으로 제한
type Note = 'C' | 'C#' | 'D' | 'D#' | 'E' | 'F' | 'F#' | 'G' | 'G#' | 'A' | 'A#' | 'B';

// 한 줄의 운지: 개방현(0), 뮤트(null), 또는 프렛 번호(양수)
type StringFret = number | 0 | null;

// 코드 보이싱 — 6현 기타 기준, 6번줄→1번줄 순서
interface ChordVoicing {
  strings: [StringFret, StringFret, StringFret, StringFret, StringFret, StringFret];
  baseFret: number;        // 다이어그램 시작 프렛
  fingers?: (number | null)[]; // 운지 손가락 번호(1~4), 선택
}

interface Chord {
  name: string;            // 예: "Cmaj7"
  root: Note;
  voicings: ChordVoicing[];
}
```
> 이 타입들은 예시다. 실제 기능에 맞게 확정하되, 유니온/튜플로 불가능한 값을 차단하는 원칙을 지킨다.

### 4. 컴포넌트 트리 & 모듈 경계 설계
- UI를 컴포넌트 트리로 그린다(부모→자식, props 흐름).
- 순수 도메인 로직(코드 계산, 진행 생성)은 컴포넌트에서 분리해 `src/lib` 또는 `src/domain`에 둔다 — 테스트하기 쉽고 재사용된다.
- 상태 관리 위치를 정한다(로컬 state vs 공유 훅/컨텍스트).

### 5. 검증 경계면 명시
QA가 검증할 데이터 계약 지점을 목록화한다. 각 항목에 생산자/소비자/기대 shape를 적는다.

| 경계면 | 생산자 | 소비자 | 계약(shape) |
|--------|--------|--------|------------|
| 코드 조회 → 다이어그램 | `getChord(name)` | `<ChordDiagram>` | `Chord` 또는 `null` |
| ... | | | |

### 6. 단계별 구현 계획
구현자가 TDD로 따라갈 수 있도록, 한 번에 구현·테스트 가능한 작은 단위로 순서대로 나열한다. 각 단계에 "무엇을 만들고", "어떤 테스트로 검증하는지"를 적는다.

## 그린필드 셋업 (빈 프로젝트일 때)
권장 스택: **Vite + React + TypeScript + Vitest + React Testing Library**.
계획에 다음을 포함한다:
- `npm create vite@latest . -- --template react-ts` 기반 초기화
- 디렉토리: `src/components/`, `src/lib/`(도메인 로직), `src/types/`, `src/hooks/`
- 테스트: Vitest + `@testing-library/react`, `src/**/*.test.ts(x)`

## 출력
`_workspace/01_architect_plan.md`에 다음 순서로 작성한다:

```markdown
# 기능: {기능명}
## 수용 기준
## 데이터 모델 (TypeScript 타입)
## 컴포넌트 트리 & 모듈 경계
## 검증 경계면
## 단계별 구현 계획
```

## 원칙
- 추측하지 말고, 모호한 요구는 질의한다. 잘못된 가정 위의 정교한 설계는 무의미하다.
- 도메인 타입을 먼저 확정한다 — 모든 UI/로직이 여기에 의존한다.
- 한 단계가 "한 호흡에 구현+테스트" 가능한 크기인지 점검한다.
